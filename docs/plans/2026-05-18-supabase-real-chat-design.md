# Supabase Real Chat Design

## Goal

Replace the mock public demo with real account login, automatic contacts, and realtime conversations between logged-in users.

## Architecture

- Use Supabase Auth with email and password.
- Use Supabase Postgres tables for profiles and messages.
- Use Supabase Realtime subscriptions for new messages.
- Keep the Next.js frontend hosted on Vercel.
- Stop using mock conversations for the public production deployment.

## Data Model

- `profiles`: one row per authenticated user.
- `messages`: one row per message with `sender_id`, `receiver_id`, `body`, `type`, `status`, and timestamps.
- Contacts are generated automatically by reading all `profiles` except the current user.
- Conversation history is generated by querying `messages` where the current user and selected contact are sender/receiver.

## Security

- Enable Row Level Security on both tables.
- Authenticated users can read profiles.
- Users can insert/update only their own profile.
- Users can read messages only when they are sender or receiver.
- Users can insert messages only as themselves.

## Deployment

- Vercel uses `NEXT_PUBLIC_DEMO_MODE=false`.
- Vercel uses public Supabase values only: `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY`.
- The Postgres connection string and service role key must never be committed or sent to the browser.
